# 1.7.0 — released 2026-05-06

> **Breaking:** legacy `CMakeSettings.json` removed — build via `CMakePresets.json` only.

## Features
- `build.bat` now offers quick clean targets so you no longer have to delete folders by hand: `build.bat clean` resets the Debug/Release outputs and `installation/` while keeping the dependency cache, `build.bat clean-deps` clears stale dependency build/subbuild folders while keeping the source downloads (handy after a generator mismatch), and `build.bat clean-all` wipes everything under `build/` and `installation/` for a true from-scratch build. A `build.bat help` target lists all available commands. See `documentation/HowToUse.md` for details. (TASK-0004 / ISSUE-0003)
- `build.bat` was rewritten to use CMake presets (`cmake --preset` / `cmake --build --preset`) and now matches Visual Studio's CMake-integration layout exactly. Per-preset binary directories (`build/x64-Debug`, `build/x64-Release`, `build/x64-Debug-Profile`, `build/x64-Release-Profile`) are shared with VS, so switching between command-line and IDE no longer triggers a regenerate. Multi-threaded compilation is enabled by default (CMAKE_BUILD_PARALLEL_LEVEL defaults to NUMBER_OF_PROCESSORS; override via env var). Visual Studio's developer environment (cl.exe, link.exe, ninja) is auto-initialized via vswhere if not already active, so the script runs from any cmd. New options: `build.bat <preset> [<preset>..]` builds specific presets; `build.bat list` enumerates available presets. Configuration knobs (`CMAKE_INSTALL_PREFIX`, `QT_DEPLOY`, `CMAKE_POLICY_VERSION_MINIMUM`, `CMAKE_CXX_FLAGS=/EHsc`) moved into `CMakePresets.json` as the single source of truth.

## Bugfixes
- First-time builds on CMake 4.x no longer fail on dependencies pinned to older CMake policy versions (notably easy_profiler v2.1.0). `build.bat` now passes `-DCMAKE_POLICY_VERSION_MINIMUM=3.5` by default, which unblocks CMake 4.x and is a no-op on earlier versions. See `documentation/CommonProblems.md` for the PowerShell quoting trap when invoking cmake manually. (TASK-0003 / ISSUE-0002)
- Successful installs no longer print spurious `"<exe> does not exist"` lines. Two-part fix: (1) the redundant duplicate `windeployqt` call was removed from `cmake/ExampleMaster.cmake`; (2) `CMAKE_{ARCHIVE,LIBRARY,RUNTIME}_OUTPUT_DIRECTORY` now embed `$<CONFIG>` so any code that captures the variable into an `install(CODE)` block ends up with a path that resolves to the actual per-config binary at install time, instead of the bare build root. Build-tree Qt deployment continues to use `$<TARGET_FILE_DIR:...>` via the POST_BUILD step. `FETCHCONTENT_BASE_DIR` is kept config-independent so dependencies aren't re-fetched per configuration. (TASK-0005 / ISSUE-0004)
- Documented a Git Bash pitfall where MSYS rewrites slash-prefixed cmake flags (e.g. `/EHsc`) into Windows paths, causing `error C1083: source file not opened: "Files/Git/EHsc"`. `documentation/CommonProblems.md` now describes the symptom and the `MSYS_NO_PATHCONV=1` workaround (or running from cmd/PowerShell). `build.bat` itself was never affected. (TASK-0006 / ISSUE-0005)
- Dependency subbuilds now inherit the project's CMake generator. Previously, regenerating the CMake cache (for example after editing a header) could fail with "Does not match the generator used previously: Ninja" for every dependency, forcing a full cache wipe to recover. The generator, platform, toolset, and make-program are now forwarded to FetchContent subbuilds so reconfigures are stable across IDE-driven and command-line builds. (TASK-0002 / ISSUE-0001)
- Switching between CMake presets (e.g. `x64-Debug` ↔ `x64-Debug-Profile` in Visual Studio) no longer wipes the shared dependency cache. Previously the cache-wipe sentinel lived inside each preset's binary directory, so the first configure of any new preset deleted dependency sources that another preset was still using, forcing a re-download and re-build. The new scheme uses two stamps: a per-preset stamp inside each binary directory plus a registry under `build/dependencies/cache/.preset_stamps/`. "Delete Cache and Reconfigure" still wipes dependencies for the preset it targets, but a brand-new preset preserves the shared cache. (Reported follow-up on ISSUE-0001 / TASK-0002.)

## Deprecations
- Removed `CMakeSettings.json` (legacy Visual Studio CMake integration). Use `CMakePresets.json` instead.
